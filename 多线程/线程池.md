# 线程池

## 1. 为什么用线程池

- 线程创建/销毁开销大，线程池复用线程，降低开销
- 控制并发数量，防止无限创建线程耗尽系统资源
- 提供任务队列、拒绝策略、线程管理等能力

---

## 2. 核心参数（ThreadPoolExecutor）

```java
new ThreadPoolExecutor(
    int corePoolSize,        // 核心线程数（常驻，不会被销毁）
    int maximumPoolSize,     // 最大线程数
    long keepAliveTime,      // 非核心线程空闲存活时间
    TimeUnit unit,           // 时间单位
    BlockingQueue workQueue, // 任务等待队列
    ThreadFactory factory,   // 线程工厂（可自定义线程名）
    RejectedExecutionHandler handler // 拒绝策略
);
```

### 参数详解

| 参数 | 说明 |
|------|------|
| corePoolSize | 核心线程数，任务少时也保持存活 |
| maximumPoolSize | 线程数上限，队列满后才会扩容到此值 |
| keepAliveTime | 超过 core 的线程空闲超时后销毁 |
| workQueue | 任务积压时放入队列等待 |

---

## 3. 任务提交流程（重点）

```
提交任务
   │
   ├─ 当前线程数 < corePoolSize    ──→ 创建核心线程执行
   │
   ├─ 当前线程数 >= corePoolSize   ──→ 放入 workQueue 排队
   │
   ├─ workQueue 已满               ──→ 创建非核心线程执行（不超过 max）
   │
   └─ 线程数达到 maximumPoolSize 且队列满  ──→ 触发拒绝策略
```

**记忆口诀：核心线程 → 队列 → 扩容到 max → 拒绝**

---

## 4. 常用阻塞队列

| 队列类型 | 特点 | 适用场景 |
|----------|------|----------|
| LinkedBlockingQueue | 无界队列（默认 Integer.MAX_VALUE） | 任务量可控时，吞吐高 |
| ArrayBlockingQueue | 有界队列，需指定容量 | 需要限制积压数量 |
| SynchronousQueue | 不存储任务，直接交给线程 | maximumPoolSize 大，快速响应 |
| PriorityBlockingQueue | 按优先级排序 | 有任务优先级需求 |

> 注意：`Executors.newFixedThreadPool` 使用的是 **LinkedBlockingQueue（无界）**，在高并发下可能导致 OOM，生产环境要用 `ThreadPoolExecutor` 手动指定有界队列。

---

## 5. 四种拒绝策略

| 策略 | 类名 | 行为 |
|------|------|------|
| 抛异常（默认） | AbortPolicy | 抛出 RejectedExecutionException |
| 调用者运行 | CallerRunsPolicy | 由提交任务的线程自己执行，起到降速作用 |
| 丢弃最旧任务 | DiscardOldestPolicy | 丢掉队列头部最老的任务，再重新提交 |
| 直接丢弃 | DiscardPolicy | 静默丢弃，不抛异常 |

**生产推荐：CallerRunsPolicy**，既不丢任务，又能反压限速。

---

## 6. Executors 工厂方法（及其坑）

```java
Executors.newFixedThreadPool(n)      // 固定线程数，无界队列 —— 队列可能OOM
Executors.newCachedThreadPool()      // 线程数无上限，队列为SynchronousQueue —— 线程可能OOM
Executors.newSingleThreadExecutor()  // 单线程，无界队列
Executors.newScheduledThreadPool(n)  // 定时任务
```

**阿里规范：禁止使用 Executors，必须手动创建 ThreadPoolExecutor 并指定有界队列。**

---

## 7. 线程池状态

```
RUNNING   → 接受新任务，处理队列任务
SHUTDOWN  → 不接受新任务（调用 shutdown()），继续处理队列剩余任务
STOP      → 不接受新任务（调用 shutdownNow()），中断正在执行的任务
TIDYING   → 所有任务终止，线程数为0，准备执行 terminated()
TERMINATED → 终止完成
```

- `shutdown()`：优雅关闭，等待已提交任务执行完
- `shutdownNow()`：立即关闭，尝试中断正在执行的任务，返回未执行的任务列表

---

## 8. 线程池参数怎么设置

### 经验公式

| 任务类型 | 建议核心线程数 |
|----------|----------------|
| CPU 密集型 | CPU 核心数 + 1 |
| IO 密集型 | CPU 核心数 × 2（或更高，根据 IO 等待比例调整） |

```java
// 获取 CPU 核心数
int cpuCores = Runtime.getRuntime().availableProcessors();
```

### 更准确的公式（Little's Law）

```
线程数 = CPU核心数 × (1 + 等待时间/计算时间)
```

IO 密集型任务（如数据库查询），等待时间远大于计算时间，线程数可以设置更大。

## 9. 代码示例

```java
// 生产环境推荐写法
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    4,                              // corePoolSize：4核机器
    8,                              // maximumPoolSize
    60, TimeUnit.SECONDS,           // 非核心线程空闲60s后销毁
    new ArrayBlockingQueue<>(200),  // 有界队列，最多积压200个任务
    new ThreadFactory() {
        private AtomicInteger count = new AtomicInteger(0);
        public Thread newThread(Runnable r) {
            Thread t = new Thread(r);
            t.setName("biz-thread-" + count.incrementAndGet());
            return t;
        }
    },
    new ThreadPoolExecutor.CallerRunsPolicy() // 调用者运行，起降速作用
);

try {
    executor.execute(() -> {
        // 业务逻辑
    });
} finally {
    // 应用关闭时优雅关闭线程池
    executor.shutdown();
}
```

## 10. 面试高频问题

**Q：线程池的核心参数有哪些？任务提交流程是怎样的？**

七个参数：corePoolSize、maximumPoolSize、keepAliveTime、unit、workQueue、threadFactory、handler。
提交流程：先用核心线程 → 核心线程满了进队列 → 队列满了扩容到 max → max 也满了触发拒绝策略。

**Q：为什么不建议用 Executors 创建线程池？**

`newFixedThreadPool` 和 `newSingleThreadExecutor` 使用无界 LinkedBlockingQueue，任务积压可能导致 OOM；
`newCachedThreadPool` 最大线程数为 Integer.MAX_VALUE，可能创建大量线程导致 OOM。

**Q：CallerRunsPolicy 有什么好处？**

由调用方线程（通常是主线程）执行任务，会阻塞调用方继续提交新任务，起到**反压/限速**的效果，既不丢任务，也能保护线程池不被压垮。

**Q：线程池中的线程数怎么设置？**

CPU 密集型设 CPU 核数 + 1；IO 密集型根据 IO 等待比例设更大的值，通常 CPU 核数 × 2，或用 Little's Law 公式计算。实际上需要压测后调整。

**Q：shutdown 和 shutdownNow 的区别？**

`shutdown()` 不再接受新任务，但等待队列中已有任务全部执行完毕再关闭；
`shutdownNow()` 立即中断正在执行的任务，返回未执行的任务列表。
