### 1. 序列化和反序列化
- 序列化：把内存中的对象转换成可以存储或传输的格式（字节流、JSON、XML 等）。
- 反序列化：把存储或传输的格式还原回内存中的对象。


- 内存对象  →  序列化  →  字节流/JSON/XML
- 字节流/JSON/XML  →  反序列化  →  内存对象

###2. 为什么需要
- 内存中的对象是"活的"，进程结束就消失了，而且不能直接通过网络传输。序列化解决两个场景：

  - 持久化：把对象存到磁盘、数据库，下次还能还原
  - 网络传输：微服务之间调用，对象需要变成字节流才能传

### Java 原生序列化 
- 实现 Serializable 接口，用 ObjectOutputStream / ObjectInputStream。
- 缺点很明显：性能差、序列化结果体积大、不跨语言，实际项目基本不用。

## 常见序列化协议
### 文本协议（可读性好，体积较大）
- JSON 是最常用的，几乎所有 HTTP 接口都用它，Jackson 和 Gson 是 Java 里最主流的库。XML 现在主要在老系统和 SOAP 协议里还能看到。

### 二进制协议（性能好，体积小，不可读）
| 协议 | 特点 | 常见场景 |
|----|----|------|
|  Protobuf  |  Google 出品，性能极好，需要写 .proto 文件定义结构  |  gRPC、跨语言 RPC    |
|  Hessian  |  无需定义文件，跨语言，性能中等  |   Dubbo 默认协议   |
|  Kryo  |  只支持 Java，性能是 Java 生态里最快之一  | Spark、缓存序列化     |
|  Thrift  | Facebook 出品，支持多语言，自带 RPC 框架   |    跨语言服务调用  |
|  MsgPack  |  类似 JSON 的二进制版，跨语言  |  对性能有要求的 JSON 替代    |

### 怎么选
- HTTP 接口对外暴露          → JSON
- 内部微服务 RPC 调用        → Protobuf 或 Hessian（看框架）
- Dubbo 项目               → Hessian（默认）或 Kryo
- gRPC 项目                → Protobuf（强制）
- Redis 缓存存对象           → Kryo 或 JSON
- 需要跨语言且性能要求高      → Protobuf


- 金融系统，服务间调用如果用 Dubbo 的话，默认就是 Hessian 序列化，对外 HTTP 接口一般是 JSON。
