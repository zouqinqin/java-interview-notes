# 编译器处理 — 面试知识点

## 一、Java 编译流程总览

```
Java 源码(.java)
      ↓  javac（前端编译器）
  字节码(.class)
      ↓  JVM 类加载
  解释执行 / JIT 编译
      ↓  热点代码触发 JIT
  本地机器码（直接执行）
```

> Java 是**半编译半解释**语言：先编译成字节码，再由 JVM 解释或 JIT 编译执行。

---

## 二、前端编译器（javac）

### 2.1 作用
将 `.java` 源文件编译为 `.class` 字节码文件，**不做任何优化**，优化工作交给 JIT。

### 2.2 编译阶段
| 阶段 | 说明 |
|------|------|
| 词法分析 | 将源码拆分为 Token（关键字、标识符、运算符等） |
| 语法分析 | 构建抽象语法树（AST） |
| 语义分析 | 类型检查、变量解析、常量折叠 |
| 语法糖脱糖 | 泛型擦除、自动装箱、foreach 展开等 |
| 字节码生成 | 将 AST 转换为 `.class` 字节码 |

### 2.3 常量折叠（面试考点）
```java
// 源码
String s = "hello" + " " + "world";

// 编译后等价于（编译期直接合并）
String s = "hello world";
```
> 编译期能确定的常量表达式，javac 会直接计算结果，减少运行时开销。

## 三、JIT 编译器（即时编译）

### 3.1 什么是 JIT？
JIT（Just-In-Time Compiler）是 JVM 运行时的**热点代码编译器**，将频繁执行的字节码编译为本地机器码，避免重复解释执行。

### 3.2 热点代码判定
JVM 通过**计数器**判断热点代码：

| 计数器 | 说明 | 默认阈值（Client/Server） |
|--------|------|--------------------------|
| 方法调用计数器 | 方法被调用的次数 | 1500 / 10000 |
| 回边计数器 | 循环体执行次数 | 13995 / 10700 |

> 超过阈值 → 触发 JIT 编译 → 编译结果缓存到 **Code Cache**

### 3.3 分层编译（Tiered Compilation）
JDK8+ 默认开启 `-XX:+TieredCompilation`，分 5 个层次：

```
Level 0：解释执行
Level 1：C1 编译（简单优化，无 profiling）
Level 2：C1 编译（有限 profiling）
Level 3：C1 编译（完整 profiling）
Level 4：C2 编译（激进优化，最高性能）
```

| 编译器 | 特点 | 适用场景 |
|--------|------|---------|
| C1（Client） | 编译快，优化少，启动快 | 短生命周期应用 |
| C2（Server） | 编译慢，优化激进，峰值性能高 | 长期运行的服务 |

### 3.4 面试常问：JIT 有哪些优化手段？

#### ① 方法内联（最重要）
```java
// 源码
int add(int a, int b) { return a + b; }
int result = add(1, 2);

// JIT 内联后（消除方法调用开销）
int result = 1 + 2;
```
> 内联是其他优化的基础，开启条件：方法体小（< 35 字节码）且调用频繁。

#### ② 逃逸分析（Escape Analysis）
分析对象的**作用域是否超出当前方法/线程**：

| 逃逸状态 | 说明 | 优化手段 |
|---------|------|---------|
| 不逃逸 | 对象只在方法内使用 | 栈上分配、标量替换 |
| 线程逃逸 | 对象只在当前线程内使用 | 消除同步锁 |
| 全局逃逸 | 对象被外部引用 | 无法优化 |

```java
// 对象不逃逸 → JIT 可以栈上分配，不进堆，减少 GC 压力
public void foo() {
    Point p = new Point(1, 2); // p 不会逃逸出 foo()
    System.out.println(p.x + p.y);
}
```

#### ③ 栈上分配
- 不逃逸的对象直接在**栈帧**上分配，方法结束自动释放，**无需 GC**
- 开启条件：`-XX:+DoEscapeAnalysis`（JDK8+ 默认开启）

#### ④ 标量替换
将对象拆解为基本类型字段，直接存在寄存器或栈上：
```java
// 原始
Point p = new Point(1, 2);
int sum = p.x + p.y;

// 标量替换后（不创建 Point 对象）
int px = 1, py = 2;
int sum = px + py;
```

#### ⑤ 锁消除
```java
// StringBuffer 的 append 有 synchronized，但 sb 不逃逸
// JIT 会自动消除这些无意义的锁
public String concat(String a, String b) {
    StringBuffer sb = new StringBuffer();
    sb.append(a);
    sb.append(b);
    return sb.toString();
}
```

## 四、语法糖（Syntactic Sugar）

> 语法糖是 javac 在编译期处理的，字节码中不存在这些语法。

### 4.1 泛型擦除
```java
// 源码
List<String> list = new ArrayList<String>();
list.add("hello");
String s = list.get(0);

// 编译后（类型信息被擦除）
List list = new ArrayList();
list.add("hello");
String s = (String) list.get(0);  // 自动插入强转
```
> **面试常问**：为什么泛型不能用基本类型？
> → 因为泛型擦除后变成 Object，而基本类型不是 Object 的子类。

### 4.2 自动装箱/拆箱
```java
Integer a = 1;        // 编译后：Integer a = Integer.valueOf(1);
int b = a;            // 编译后：int b = a.intValue();
```
> **面试陷阱**：`Integer.valueOf()` 有缓存（-128 ~ 127），超出范围 `==` 比较为 false。

### 4.3 foreach 展开
```java
// 数组的 foreach
for (int x : arr) { }
// 编译后
for (int i = 0; i < arr.length; i++) { int x = arr[i]; }

// 集合的 foreach
for (String s : list) { }
// 编译后
Iterator<String> it = list.iterator();
while (it.hasNext()) { String s = it.next(); }
```

### 4.4 try-with-resources
```java
// 源码
try (InputStream in = new FileInputStream("file")) {
    // ...
}

// 编译后（自动插入 close() 和异常处理）
InputStream in = new FileInputStream("file");
Throwable primary = null;
try {
    // ...
} catch (Throwable t) {
    primary = t; throw t;
} finally {
    if (primary != null) {
        try { in.close(); } catch (Throwable t) { primary.addSuppressed(t); }
    } else { in.close(); }
}
```

### 4.5 其他语法糖汇总

| 语法糖 | 编译后等价 |
|--------|-----------|
| `var`（JDK10+） | 推断具体类型 |
| `switch` 字符串 | `hashCode()` + `equals()` |
| 内部类 | 独立的 `$` 命名 class 文件 |
| 枚举 `enum` | 继承 `Enum` 的 final 类 |
| 注解 `@interface` | 继承 `Annotation` 的接口 |
| 可变参数 `...` | 数组 `T[]` |

## 五、AOT 编译（Ahead-Of-Time）

| | JIT | AOT |
|--|-----|-----|
| 编译时机 | 运行时 | 运行前 |
| 启动速度 | 慢（需预热） | 快 |
| 峰值性能 | 高（有 profiling 优化） | 较低 |
| 代表实现 | HotSpot C2 | GraalVM Native Image |

> **面试常问**：微服务/Serverless 场景为什么用 GraalVM？
> → AOT 编译成原生可执行文件，启动时间从秒级降到毫秒级，内存占用更小。

## 六、Code Cache（代码缓存）

- JIT 编译后的机器码存放在 **Code Cache** 中（属于堆外内存）
- 默认大小：JDK8 约 240MB（64位 Server 模式）
- Code Cache 满了 → JIT 停止编译 → 退化为解释执行 → 性能骤降

```bash
# 相关参数
-XX:InitialCodeCacheSize=32m    # 初始大小
-XX:ReservedCodeCacheSize=256m  # 最大大小
-XX:+UseCodeCacheFlushing       # 满了时清理旧代码（默认开启）
```

> **排查手段**：`jcmd <pid> Compiler.codecache` 查看 Code Cache 使用情况

## 七、面试高频问题汇总

### Q1：解释执行和编译执行的区别？
- **解释执行**：逐条读取字节码并翻译执行，启动快但运行慢
- **编译执行**：JIT 将热点字节码编译为机器码，运行快但有编译开销
- JVM 默认**混合模式**：先解释，热点代码 JIT 编译

### Q2：为什么 Java 启动慢，但运行一段时间后变快？
→ JIT 预热：初期解释执行，热点代码被 JIT 编译后性能大幅提升（Warm-up 过程）

### Q3：泛型擦除会带来什么问题？
1. 运行时无法获取泛型类型（`List<String>` 和 `List<Integer>` 运行时相同）
2. 不能用基本类型作为泛型参数
3. 泛型数组创建受限（`new T[]` 不合法）

### Q4：逃逸分析一定有效吗？
不一定。以下情况逃逸分析失效：
- 对象被赋值给静态变量
- 对象作为方法返回值
- 对象被传入其他方法且无法内联

### Q5：如何验证 JIT 是否生效？
```bash
-XX:+PrintCompilation          # 打印 JIT 编译的方法
-XX:+UnlockDiagnosticVMOptions
-XX:+PrintInlining             # 打印内联信息
-XX:+PrintEscapeAnalysis       # 打印逃逸分析结果
```
