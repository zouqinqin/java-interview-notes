## 1.1 引用计数法
1. 只要对象被引用，计数 +1，如果某个变量不再引用它了，计数 -1，如果最后计数为 0，就可以被回收
2. 有一个弊端：当 A 对象引用 B，B 又引用 A，只要它们相互引用，计数永远不为 0，就会造成**内存泄漏**，因此 Java 并未采用此方案

## 1.2 可达性分析算法
1. Java 虚拟机中的垃圾回收器采用**可达性分析**来探测所有存活的对象
2. 扫描堆中的对象，看是否能够沿着 GC Root 对象作为起点的引用链找到该对象，找不到，便表示可以回收

- 哪些对象可以作为 GC Root 呢？
  - **虚拟机栈**中引用的对象（当前正在执行的方法中的局部变量所引用的对象）
  - **方法区中类静态属性**引用的对象（static 修饰的变量所引用的对象）
  - **方法区中常量**引用的对象（final 修饰的常量所引用的对象）
  - **本地方法栈**中 JNI（Native 方法）引用的对象
  - 可以用 Eclipse Memory Analyzer（MAT）工具来查看 GC Root

## 1.3 四种引用

> 引用强度：强引用 > 软引用 > 弱引用 > 虚引用

1. **强引用（Strong Reference）**
   - 最常见的引用方式，如 `Object obj = new Object()`
   - 只要强引用存在，GC 永远不会回收该对象
   - 只有当找不到任何 GC Root 能到达该对象时，才可以被回收

2. **软引用（Soft Reference）**
   - 用 `SoftReference<T>` 包装对象
   - 在垃圾回收时，**内存不足**时才会被回收（前提是没有强引用指向它）
   - 适合做**内存敏感的缓存**，内存够用时保留，内存紧张时自动释放

3. **弱引用（Weak Reference）**
   - 用 `WeakReference<T>` 包装对象
   - 只要发生垃圾回收，**无论内存是否充足**，都会被回收（前提是没有强引用指向它）
   - 适合做**非必要缓存**，如 `ThreadLocal` 中的 key 就是弱引用

4. **软引用和弱引用**可以配合**引用队列（ReferenceQueue）** 来使用
   - 当被引用的对象被回收后，软/弱引用对象本身会被加入引用队列
   - 可以通过遍历引用队列，清理掉这些已失效的引用对象，避免引用对象本身占用内存

5. **虚引用（Phantom Reference）**
   - 用 `PhantomReference<T>` 包装对象，**必须配合引用队列使用**
   - 无法通过虚引用获取对象（`get()` 永远返回 null）
   - 作用：在对象被回收时，收到一个系统通知，用于做**资源清理**（如 NIO 的直接内存释放）

6. **终结器引用（Finalizer Reference）**
   - 无需手动使用，由 JVM 内部使用
   - 当对象重写了 `finalize()` 方法时，GC 不会立即回收该对象，而是将其加入终结器引用队列
   - 由一个低优先级的 Finalizer 线程执行 `finalize()` 方法后，下次 GC 才真正回收
   - 由于执行时机不确定，**不推荐**使用 `finalize()` 做资源释放
